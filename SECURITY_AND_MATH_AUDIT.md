# Аудит безопасности и математики

## 1. Проверка математики

### Константы (config.py)
- **MATRIX_PRICES**: M1=$10, M2=$20, M3=$40, M4=$80.
- **MATRIX_INCOME**: валовой доход при полном заполнении (4 места): 36, 72, 144, 288.
- **MATRIX_BONUS_PER_PLACE**: чистая выплата за одно место (доход минус реинвест), в долларах:
  - M1: (36−10)/4 = 6.50
  - M2: (72−20)/4 = 13.00
  - M3: (144−40)/4 = 26.00
  - M4: (288−80)/4 = 52.00
- **REINVEST_CONFIG**: при закрытии M1 открывается M1 ($10); M2 → M1,M2 ($20); M3 → M1,M2,M3 ($40); M4 → M1,M2,M3,M4 ($80).

При импорте конфига выполняется `_verify_math()`: для каждого уровня проверяется  
`4 * MATRIX_BONUS_PER_PLACE[L] == MATRIX_INCOME[L] - REINVEST_CONFIG[L]['cost']`.

### Комиссия и распределение
- **10%** от каждой оплаты (регистрация, докупка) → организатор (admin_fee, SYSTEM_USER_ID).
- **90%** концептуально идут «в сеть» и выплачиваются по мере заполнения позиций 4–7 в матрицах (matrix_bonus). Сумма выплат не может превысить 90% оборота; разница — «в пути», пока не все места заполнены.

### Защита от ошибочных начислений
- Все суммы в операциях с балансом и бонусами округляются до 2 знаков (`_round_money`).
- Бонус начисляется только при добавлении **новой** позиции 4, 5, 6 или 7. В БД действует `UniqueConstraint("matrix_id", "position")`, поэтому одна и та же позиция не может быть заполнена дважды → двойное начисление за одно место невозможно.
- Уровни матриц: только 1, 2, 3, 4; дубликаты уровней в запросе отбрасываются (один пользователь — одна активная матрица уровня L).
- **Докупка по порядку**: при докупке нельзя купить уровень L, не имея уровней 1..L−1 (нельзя купить M3 без M1 и M2, M4 — без M1, M2, M3). Проверка: `_can_purchase_levels()`.
- **Размещение при докупке**: при докупке матриц пользователь размещается в цепочке реферера по каждому докупленному уровню (тот же алгоритм, что при регистрации). Матрицы по уровням независимы — пользователь может оказаться в разных ветках на M1–M2 и на M3–M4.

## 2. Уязвимости и ограничения

### Без аутентификации
- API не требует логина. Любой может вызывать регистрацию, докупку, пополнение баланса. Приложение предназначено для симуляции/теста; для продакшена нужна авторизация и разграничение прав.

### add_funds
- Позволяет зачислить произвольную сумму любому пользователю (кроме системного). В API добавлена верхняя граница 1 000 000 для ограничения злоупотреблений. В продакшене эту функцию следует отключить или ограничить админ-ролью.

### Ввод
- Уровни матриц проверяются (1–4), пустой список уровней отклоняется.
- `referrer_id` проверяется на существование и не может быть системным.
- Имена пользователей уникальны; при регистрации проверяется занятость.

### База данных
- Используется ORM (SQLAlchemy), сырых SQL-запросов нет → риск SQL-инъекций отсутствует.

### Целостность баланса
- Эндпоинт `GET /api/admin/verify` проверяет, что для каждого пользователя (включая системного) баланс совпадает с суммой транзакций (с допуском 0.01). Рекомендуется периодически вызывать проверку после массовых операций.

## 3. Внесённые изменения (аудит)

1. **Округление**: все зачисления/списания и бонусы округляются до 2 знаков; баланс при чтении тоже округляется.
2. **Валидация уровней**: при регистрации и докупке уровни приводятся к списку из 1–4 без дубликатов; пустой список даёт ошибку.
3. **Проверка констант**: при старте приложения выполняется `_verify_math()` в config.
4. **admin_verify**: проверка баланса включена и для системного пользователя.
5. **add_funds**: отклонение `user_id <= 0` и системного пользователя; верхний предел суммы 1 000 000.
6. **Ошибки сервиса**: при невалидных уровнях сервис регистрации выбрасывает `ValueError`; API возвращает 400 с сообщением.

---

## 4. Рекомендации для продакшена

- Ввести аутентификацию (JWT/session) и проверку прав на админ-действия.
- Ограничить или убрать `add_funds`; пополнение — только через платёжный контур с проверкой.
- Рассмотреть использование `Decimal` для денежных полей и расчётов.
- Логировать все операции, меняющие баланс и матрицы, для аудита.
